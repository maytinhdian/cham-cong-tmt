# ZKTeco PUSH Important API Reference

Source: `zkteco-sdk.pdf` - Attendance PUSH Communication Protocol, PUSH SDK 2.4.1 / Doc 3.9.

This file records the API subset most relevant to TMT Time Attendance.

## Conventions

- Device is the HTTP client.
- Server response body is plain text.
- `SN` is required for nearly all requests and identifies the device serial number.
- `${HT}` means tab.
- `${LF}` means newline.
- Command IDs are generated by the server and must be echoed by the device in command replies.

## Initialize Options

This is the first request a device sends after its ADMS/PUSH server IP and port are configured. The
protocol family describes two initialization paths:

- Registered device: server recognizes the serial number and returns the registration/configuration
  information needed to complete the connection.
- Unregistered device: some newer/security-oriented flows ask the device to register first, then
  download configuration after registration succeeds.

For the current TMT attendance-log scope, the app uses the simpler attendance PUSH behavior: it
matches `SN` to `attendance_devices.code`, auto-creates an unknown device record, and immediately
returns the option block below. Add a stricter registration handshake only if a real target device
requires it.

Device request:

```http
GET /iclock/cdata?SN=${SerialNumber}&options=all&pushver=${PushVersion}&language=${Language}&pushcommkey=${Key}
```

Required query:

- `SN`: device serial number.
- `options=all`: asks server for configuration.

Optional query:

- `pushver`: device-supported PUSH protocol version.
- `language`: device language number. Vietnamese is listed as `86`; English is `69`.
- `pushcommkey`: optional binding/auth ciphertext used by some deployments.

Server response:

```text
GET OPTION FROM: ${SerialNumber}
ATTLOGStamp=None
OPERLOGStamp=None
ATTPHOTOStamp=None
ErrorDelay=30
Delay=10
TransTimes=00:00
TransInterval=1
TransFlag=TransData AttLog
TimeZone=7
Realtime=1
Encrypt=None
ServerVer=2.4.1
PushProtVer=2.4.1
```

Important response keys:

- `ATTLOGStamp`: last attendance-log timestamp marker. Use `None` or `0` when forcing reupload.
- `ErrorDelay`: reconnect delay after network failure, recommended `30` to `300` seconds.
- `Delay`: normal polling interval for `/iclock/getrequest`, recommended `2` to `60` seconds.
- `TransTimes`: scheduled upload times in `HH:mm`, separated by semicolons.
- `TransInterval`: interval upload cadence in minutes.
- `TransFlag`: data types allowed for automatic upload. Use `TransData AttLog` for attendance logs.
- `TimeZone`: server timezone. Use `7` for Vietnam.
- `Realtime`: `1` means upload new records immediately.
- `Encrypt`: `None` or `1`. Start with `None`.
- `PushProtVer`: server protocol version.

## Upload Attendance Logs

Device request:

```http
POST /iclock/cdata?SN=${SerialNumber}&table=ATTLOG&Stamp=${Stamp}
Content-Type: text/plain

${Pin}${HT}${Time}${HT}${Status}${HT}${Verify}${HT}${Workcode}${HT}${Reserved}${HT}${Reserved}
${Pin}${HT}${Time}${HT}${Status}${HT}${Verify}${HT}${Workcode}${HT}${Reserved}${HT}${Reserved}
```

Required query:

- `SN`: device serial number.
- `table=ATTLOG`: uploaded data is attendance records.

Optional query:

- `Stamp`: latest upload timestamp marker.

Record fields:

- `Pin`: device user code.
- `Time`: verification time, `YYYY-MM-DD HH:mm:ss`.
- `Status`: attendance status code.
- `Verify`: verification method code.
- `Workcode`: optional work code.
- `Reserved`: reserved field.
- `Reserved`: reserved field.

Server success response:

```text
OK: ${ProcessedCount}
```

Project handling:

- Match `SN` to `attendance_devices.code`.
- Match `Pin` to `attendance_device_user_maps.device_user_code`.
- Save into `raw_attendance_logs`.
- Keep original line or parsed extra fields in metadata where practical.

## Get Server Command

Device request:

```http
GET /iclock/getrequest?SN=${SerialNumber}
```

Server response when no command is pending:

```text
OK
```

Server response when commands are pending:

```text
C: ${CmdID}: ${CmdDesc} ${Payload}
```

Multiple commands may be separated by newline.

General command format:

```text
C: ${CmdID}: ${CmdDesc}${SP}${XXX}
```

Fields:

- `CmdID`: server-generated command ID, max 16 characters.
- `CmdDesc`: command type, such as `LOG`, `CHECK`, `DATA`, `SET`, `RELOAD OPTIONS`.
- Payload depends on command type.

## Command Reply

Device request:

```http
POST /iclock/devicecmd?SN=${SerialNumber}
Content-Type: text/plain

ID=${CmdID}&Return=${ReturnCode}&CMD=${CommandName}
```

Multiple reply rows may be separated by newline.

Server success response:

```text
OK
```

Common return code:

- `0`: success.

Other important return codes from Appendix 1:

- `-1`: parameter is incorrect.
- `-3`: read/write error.
- `-10`: PIN does not exist on device.
- `-1001`: limited capacity.
- `-1002`: not supported by equipment.
- `-1003`: command execution timeout.
- `-1005`: equipment is busy.
- `-1008`: failed to get server data.

## Command: Upload New Logs Now

Server command:

```text
C: ${CmdID}: LOG
```

Meaning:

Ask the device to immediately check whether new data exists and transmit it to the server.

Expected device reply:

```text
ID=${CmdID}&Return=${ReturnCode}&CMD=LOG
```

## Command: Check / Reupload Based On Options

Server command:

```text
C: ${CmdID}: CHECK
```

Meaning:

Ask the device to reread initialization options and reupload data based on timestamp settings. Setting a timestamp such as `ATTLOGStamp=0` can force reupload behavior on supported devices.

Expected device reply:

```text
ID=${CmdID}&Return=${ReturnCode}&CMD=CHECK
```

## Command: Query Attendance Logs By Time Range

Server command:

```text
C: ${CmdID}: DATA QUERY ATTLOG StartTime=${YYYY-MM-DD HH:mm:ss}${HT}EndTime=${YYYY-MM-DD HH:mm:ss}
```

Meaning:

Ask the device to upload attendance records in the requested period. Upload still uses the `POST /iclock/cdata?table=ATTLOG` path.

Expected device reply:

```text
ID=${CmdID}&Return=${ReturnCode}&CMD=DATA
```

## Command: Set One Device Option

Server command:

```text
C: ${CmdID}: SET OPTION ${Key}=${Value}
```

Meaning:

Set a single configuration key on the device.

Expected device reply:

```text
ID=${CmdID}&Return=${ReturnCode}&CMD=SET OPTION
```

## Command: Reload Device Options

Server command:

```text
C: ${CmdID}: RELOAD OPTIONS
```

Meaning:

Ask the device to reload configuration information.

Expected device reply:

```text
ID=${CmdID}&Return=${ReturnCode}&CMD=RELOAD OPTIONS
```

## Heartbeat / Ping

Some devices call heartbeat-compatible endpoints while polling or uploading.

Project endpoint:

```http
GET /iclock/ping?SN=${SerialNumber}
POST /iclock/ping?SN=${SerialNumber}
```

Recommended response:

```text
OK
```

Project handling:

- Touch or create the device by `SN`.
- Update connection status / last seen time.

## Remote Attendance

Device request:

```http
GET /iclock/cdata?SN=${SerialNumber}&table=RemoteAtt&PIN=${Pin}
```

Meaning:

Device asks the server for temporary user information when a user punches remotely and is not stored on the device.

Possible server response when user exists:

```text
DATA UPDATE USERINFO PIN=${Pin}${HT}Name=${Name}${HT}Passwd=${Password}${HT}Card=${Card}${HT}Grp=${Group}${HT}TZ=${TimezoneGroup}${HT}Pri=${Privilege}
DATA UPDATE FINGERTMP PIN=${Pin}${HT}FID=${FingerId}${HT}Size=${Size}${HT}Valid=${Valid}${HT}TMP=${Template}
```

Project status:

- Not required for first attendance-log integration.
- Keep as later scope if HR needs remote punching for unsynced users.

## BIODATA / Unified Biometric Templates

The protocol supports upload, query, update, delete, and clear operations for unified biometric templates.

Device uploads template to server:

```http
POST /iclock/cdata?SN=${SerialNumber}&table=BIODATA&Stamp=${Stamp}
Content-Type: text/plain

BIODATA Pin=${Pin}${HT}No=${No}${HT}Index=${Index}${HT}Valid=${Valid}${HT}Duress=${Duress}${HT}Type=${Type}${HT}MajorVer=${MajorVer}${HT}MinorVer=${MinorVer}${HT}Format=${Format}${HT}Tmp=${Template}
```

Important fields:

- `Pin`: employee/device user number.
- `No`: biometric individual number, such as finger number or palm side.
- `Index`: template number for the biometric individual.
- `Valid`: `0` invalid, `1` valid.
- `Duress`: `0` normal, `1` duress.
- `Type`: biometric type. Examples include `1` fingerprint, `2` face, `6` palmprint, `7` finger vein, `8` palm, `9` visible light face.
- `MajorVer` / `MinorVer`: algorithm version.
- `Format`: template format. For fingerprint, examples include `0` ZK, `1` ISO, `2` ANSI.
- `Tmp`: template payload. Raw binary templates are base64 encoded.

Server issues template to device:

```text
C:${CmdID}:DATA UPDATE BIODATA Pin=${Pin}${HT}No=${No}${HT}Index=${Index}${HT}Valid=${Valid}${HT}Duress=${Duress}${HT}Type=${Type}${HT}MajorVer=${MajorVer}${HT}MinorVer=${MinorVer}${HT}Format=${Format}${HT}Tmp=${Template}
```

Server queries templates from device:

```text
C:${CmdID}:DATA QUERY BIODATA Type=${Type}
C:${CmdID}:DATA QUERY BIODATA Type=${Type}${HT}PIN=${Pin}
C:${CmdID}:DATA QUERY BIODATA Type=${Type}${HT}PIN=${Pin}${HT}No=${No}
```

Server deletes templates from device:

```text
C:${CmdID}:DATA DELETE BIODATA Pin=${Pin}
C:${CmdID}:DATA DELETE BIODATA Pin=${Pin}${HT}Type=${Type}
C:${CmdID}:DATA DELETE BIODATA Pin=${Pin}${HT}Type=${Type}${HT}No=${No}
```

Server clears unified templates:

```text
C:${CmdID}:CLEAR BIODATA
```

Project status:

- This is future scope.
- Implement after real-device testing confirms which `Type`, algorithm versions, formats, and payloads are produced by the target ZKTeco model.
- Cross-device biodata sync should be server-mediated: receive from source device, store in app database, then issue `DATA UPDATE BIODATA` commands to selected target devices.

## Currently Mapped App Endpoints

The Laravel app already follows this subset:

- `GET /iclock/cdata` for initialization.
- `POST /iclock/cdata?table=ATTLOG` for attendance upload.
- `GET|POST /iclock/getrequest` for command polling.
- `GET|POST /iclock/ping` for heartbeat.
- `POST /iclock/devicecmd` for command acknowledgements.

Related operator page:

- `GET /pages/attendance/push-receiver` shows the receiver monitor for authenticated users with
  `attendance.raw_logs.view`.
- The page does not receive device HTTP traffic itself. Physical machines still post to `/iclock/*`.
- Operators with `attendance.devices.manage` can queue a `LOG` command from the page; the device
  receives it on the next `/iclock/getrequest` poll.

Implementation note:

- Queued command IDs should be compact alphanumeric values. The protocol describes command IDs as server-generated values with a maximum length of 16 characters.
